**Стек:** React 18 + Vite + Tailwind CSS + Capacitor 8 (iOS/Android) + Supabase

## 1. Пользовательские сценарии

### 1.1 Онбординг (первый запуск нативного приложения)

**Happy path:**
1. Пользователь впервые открывает приложение на iOS/Android
2. Отображаются 4 слайда онбординга (`/onboarding/1-4.jpg`)
3. Можно свайпать между слайдами или нажимать "Далее"
4. На последнем слайде кнопка "Начать" завершает онбординг
5. В `localStorage` сохраняется `onboarding_done=true`

**Edge cases:**
- На вебе онбординг не показывается (`Capacitor.isNativePlatform()` = false)
- Повторное открытие приложения — онбординг не появляется (проверка `localStorage`)
- Кнопка "Пропустить" доступна на всех слайдах кроме последнего

### 1.2 Главная страница (MainPage — `/`)

**Happy path:**
1. Загружаются все активные подопечные из Supabase
2. Отображаются категории с количеством подопечных
3. Показывается общий прогресс сбора (собрано / цель)
4. Список фондов-партнёров с горизонтальной прокруткой
5. Первые 10 подопечных отображаются в масонри-сетке (2 колонки)
6. Нажатие на категорию ведёт на `/feed?category=X`
7. Нажатие на подопечного открывает CharityModal

**Edge cases:**
- Переход по старой ссылке `/?beneficiary=X` — редирект на `/feed?beneficiary=X`
- Поиск: пустой запрос — dropdown не отображается; ввод текста — до 8 результатов
- Закрытие dropdown при клике вне области поиска
- Более 10 подопечных — кнопка "Показать всех подопечных"
- На десктопе отображается отдельная версия с другим layout
- Карточка подписки: минимальная сумма 100 тг, проверка номера 11 цифр
- Кнопки скачивания приложения не показываются в нативном приложении

### 1.3 Лента подопечных (HomePage — `/feed`)

**Happy path:**
1. Отображаются CategoryTabs (горизонтальный скролл)
2. Фильтрация по категории и городу
3. Карточки подопечных в cards-grid
4. Нажатие на карточку — CharityModal
5. На карточке кнопки "Помочь" и "Поделиться"

**Edge cases:**
- Переход с параметром `?category=X` — автопереключение города на "Вся страна"
- Переход с параметром `?beneficiary=X` — прямая загрузка подопечного по ID (без фильтра города)
- Параметр `?donated=kaspi` — показывается popup "Спасибо" с информацией о счёте
- Пустой результат — сообщение "Нет подопечных" с указанием города/категории
- Выбор города: если не выбран, через 15 секунд появляется CitySelectionModal
- Фильтр "Вся страна" скрывает карточки с "Донорство" в заголовке

### 1.4 Детальная карточка подопечного (CharityModal)

**Happy path:**
1. Полноэкранная модалка снизу (мобайл) или по центру (десктоп)
2. Галерея медиа (изображения + видео) со свайпом
3. Кнопки навигации влево/вправо для медиа
4. Название, описание, фонд-партнёр, прогресс сбора
5. Ссылка на документы (если есть)
6. Кнопки "Помочь" и "Поделиться"

**Edge cases:**
- Drag-to-close: свайп вниз > 100px закрывает модалку с анимацией
- YouTube видео: парсинг URL через regex для embedded iframe
- Не-YouTube видео: нативный `<video>` элемент
- Если media = 1, кнопки навигации скрыты
- Scroll lock: `document.body.style.overflow = 'hidden'` при открытии

### 1.5 Оплата (PaymentModal)

**Happy path:**
1. Выбор суммы из пресетов (500, 1000, 2000, 5000) или ввод своей
2. Выбор способа оплаты (Kaspi / Банковская карта — скоро)
3. Ввод номера телефона (если Kaspi и нет сохранённого)
4. Отправка заявки в `kaspi_payment_requests`
5. Upsert в таблицу `visitors` (привязка phone к visitor_id)
6. Редирект на `/feed?donated=kaspi` с popup "Спасибо"

**Edge cases:**
- Повторный пользователь: телефон подтягивается из предыдущих оплат
- Номер должен быть 11 цифр и начинаться с 7
- Защита от двойного нажатия: `isSubmitting` flag
- Банковская карта показывается но отключена ("скоро")
- Пустая сумма — кнопка disabled
- Drag-to-close аналогично CharityModal

### 1.6 Профиль (ProfilePage — `/profile`)

**Happy path (неавторизованный):**
1. Показывается placeholder с призывом сделать пожертвование
2. Кнопки авторизации: Google OAuth и Email Magic Link
3. Если есть незавершённые заявки — они показываются (Формируется / Ожидает оплаты)

**Happy path (авторизованный, с оплатами):**
1. Аватар, имя (можно менять), общая сумма помощи
2. Группировка пожертвований по категориям
3. Горизонтальный скролл карточек в каждой категории
4. Кнопка "+ Добавьте подопечных" для пустых категорий

**Edge cases:**
- Объединение visitor_id: по auth_user_id, по телефону, по payment_requests
- Автоматическое имя из Google OAuth
- Popup "Спасибо" при новых confirmed (paid) оплатах
- Загрузка аватара в Supabase Storage (`images/avatars/`)
- Ограничение имени 20 символов
- Заявки старше 24 часов не показываются в "Формируется" и "Ожидает оплаты"

### 1.7 Настройки профиля (ProfileSettingsPage — `/profile/settings`)

**Happy path:**
1. Статистика: общая сумма помощи, количество подопечных
2. Номер телефона: отображение, смена
3. Авторизация: вход/выход Google/Email
4. Удаление аккаунта

**Edge cases:**
- Удаление аккаунта: `localStorage.clear()`, отвязка auth_user_id от visitors
- Номер телефона маскируется: `+7 XXX *** ** XX`
- Без оплат — призыв сделать пожертвование для верификации

### 1.8 Страница фонда (FundDetailPage — `/fund/:name`)

**Happy path:**
1. Загрузка фонда по имени из URL (decodeURIComponent)
2. Логотип, описание (с "Ещё"), BIN, дата, город
3. Социальные ссылки (сайт, instagram, facebook, whatsapp)
4. Карточка подписки на ежемесячную помощь
5. Список подопечных фонда

**Edge cases:**
- Фонд не найден — сообщение + кнопка "Назад"
- Длинное описание — clamp 5 строк с кнопкой "Ещё"
- WhatsApp ссылка очищается от не-цифр: `https://wa.me/${cleaned}`

### 1.9 Отчёты (ReportsPage — `/reports`)

**Happy path:**
1. Загрузка подопечных с `collection_status = 'reported'`
2. Карточки: фото, заголовок, категория, сумма, дата
3. Нажатие — ReportModal с галереей фото/видео

### 1.10 Подписка на ежемесячную помощь

**Happy path (MainPage / FundDetailPage):**
1. Выбор суммы из пресетов (1000, 3000, 5000, 10000) или своя
2. Нажатие "Помогать ежемесячно" → модалка выбора оплаты
3. Kaspi → ввод телефона → отправка в `fund_subscriptions`
4. Шаг "done" → подтверждение

**Edge cases:**
- Минимальная сумма 100 тг
- Телефон 11 цифр
- visitor_id из localStorage

### 1.11 Авторизация

**Happy path (Google OAuth):**
1. Веб: редирект на Google → возврат на `/profile`
2. Нативное: `Browser.open()` → `shanyrak://auth` deep link → `setSession()`

**Happy path (Email Magic Link):**
1. Ввод email → отправка OTP через Supabase
2. Пользователь кликает ссылку → AuthCallbackPage → setSession → `/profile`

**Edge cases:**
- Native app resume: проверка сессии при возврате в foreground
- `appUrlOpen`: парсинг hash fragment для access_token/refresh_token
- Линковка visitor: auth_user_id привязывается к текущему visitor_id
- Линковка по телефону: все visitors с тем же телефоном обновляются

### 1.12 Административная панель

**Happy path:**
1. `/admin` — логин email/password
2. `/admin/dashboard` — табы: Подопечные, Завершенные, Отчёты, Фонды, Kaspi, Подписки, Пуши
3. CRUD подопечных, фондов
4. Управление Kaspi запросами: new → invoice_sent → paid / unpaid
5. Управление подписками: active, paused, cancelled
6. Отправка push-уведомлений

**Edge cases:**
- Проверка admin email в `ADMIN_EMAILS` массиве
- Проверка сессии при загрузке dashboard
- Автоматическое истечение заявок > 24ч (визуально в отдельном табе "Просроченные")

## 2. Найденные баги и проблемы

### 2.1 Критические баги

**BUG-001: Отрицательное значение "Осталось собрать" при перевыполнении цели**
- **Файл:** `/src/components/CharityCard.jsx`, строка 10
- **Описание:** `const remainingAmount = data.target - data.raised;` — если `raised > target`, отображается отрицательное число
- **Fix:** Добавить `Math.max(0, ...)` (как сделано в CharityModal строка 18, но не в CharityCard)

**BUG-002: PaymentPage (`/payment`) — заглушка без реальной интеграции**
- **Файл:** `/src/pages/PaymentPage.jsx`, строка 29
- **Описание:** `handlePayment()` просто делает `alert()` и навигацию. Это orphan-страница, не связанная с реальной оплатой. Маршрут доступен пользователям, но ничего не делает.

**BUG-003: CitySelectionModal — кнопка закрытия делает `window.location.reload()`**
- **Файл:** `/src/components/CitySelectionModal.jsx`, строка 16
- **Описание:** Закрытие модалки не через callback а через полную перезагрузку страницы — потеря состояния приложения.

**BUG-004: Иконка "edit" и "trash-2" не зарегистрированы в Icon.jsx**
- **Файл:** `/src/components/admin/BeneficiaryList.jsx`, строки 153, 159
- **Описание:** Используются иконки `edit` и `trash-2`, но в `Icon.jsx` нет таких маппингов. Будет `console.warn` и отсутствие иконок.

**BUG-005: Иконка "bar-chart-2" не зарегистрирована**
- **Файл:** `/src/pages/DocumentsPage.jsx`, строки 11, 17, 23
- **Описание:** Иконка `bar-chart-2` для отчётов не добавлена в `Icon.jsx`.

**BUG-006: Иконка "alert-circle" и "inbox" не зарегистрированы**
- **Файл:** `/src/pages/PartnerFundsPage.jsx`, строки 24, 52
- **Описание:** Не добавлены в маппинг `Icon.jsx`.

**BUG-007: className содержит мусорный текст `-check`**
- **Файл:** `/src/components/CharityCard.jsx`, строка 61; `/src/components/CharityModal.jsx`, строка 258
- **Описание:** `className="-check text-[var(--primary-color)]"` — строка `-check` является остатком от переименования иконки `shield-check` → `shield`. Не влияет на функциональность, но замусоривает DOM.

**BUG-008: `useEffect` без зависимостей для `onCityChange`**
- **Файл:** `/src/pages/HomePage.jsx`, строка 21-25
- **Описание:** `useEffect` использует `onCityChange` но не включает его в массив зависимостей `[]`. При изменении `onCityChange` (что маловероятно, но возможно) эффект не перезапустится. Linter warning.

**BUG-009: `window.location.search` вместо `searchParams` для начального состояния**
- **Файл:** `/src/pages/HomePage.jsx`, строки 15-16, 21, 69
- **Описание:** Используется `new URLSearchParams(window.location.search)` вместо хуков React Router. Может привести к рассинхронизации при SPA-навигации.

### 2.2 Средние баги

**BUG-010: Дублирование getCategoryName функции в 5 файлах**
- **Файлы:** `HomePage.jsx`, `MainPage.jsx`, `FundDetailPage.jsx`, `ReportsPage.jsx`, `BeneficiaryList.jsx`
- **Описание:** Одна и та же функция дублирована, при этом в `FundDetailPage.jsx` отсутствует категория `social`. Несоответствие данных.

**BUG-011: Нет проверки минимальной суммы в PaymentModal**
- **Файл:** `/src/components/PaymentModal.jsx`, строка 193
- **Описание:** Проверка `amount <= 0`, но нет минимума. Можно отправить заявку на 1 тг. В других местах (подписки) минимум 100 тг.

**BUG-012: formatPhone не форматирует при вводе — отображает raw digits**
- **Файл:** `/src/components/PaymentModal.jsx`
- **Описание:** Когда savedPhone установлен, телефон показывается отформатированным, но для нового пользователя поле ввода показывает `+7`, а onChange получает raw цифры, что путает.

**BUG-013: Race condition при быстрой смене категории/города**
- **Файл:** `/src/hooks/useBeneficiaries.js`
- **Описание:** Хотя `cancelled` flag есть, задержка 50ms (`setTimeout`) может привести к неконсистентному состоянию если пользователь быстро переключает фильтры.

**BUG-014: `collection_status: 'active'` хардкодится при сохранении в админке**
- **Файл:** `/src/components/admin/BeneficiaryFormModal.jsx`, строка 86
- **Описание:** При редактировании завершённого/отчётного подопечного статус принудительно меняется на `active`.

**BUG-015: dataCache.js не используется нигде**
- **Файл:** `/src/utils/dataCache.js`
- **Описание:** Утилита кеширования в localStorage создана, но нигде не импортируется. Мёртвый код.

**BUG-016: charityData.js — не проверялся, но скорее всего мёртвый код**
- **Файл:** `/src/utils/charityData.js`
- **Описание:** Не импортируется ни одним компонентом.

### 2.3 Незначительные баги

**BUG-017: CSS portrait lock через transform — ломает layout**
- **Файл:** `/src/index.css`, строки 113-124
- **Описание:** `transform: rotate(-90deg)` на `<html>` в landscape — агрессивный подход, может сломать модалки и absolute-позиционированные элементы.

**BUG-018: Дублирование кода More Menu в MainPage и ProfilePage**
- **Описание:** Два идентичных компонента "Дополнительно" меню. Должен быть переиспользуемый компонент.

**BUG-019: ProfileSettingsPage — formatPhoneDisplay маскирует номер при редактировании**
- **Файл:** `/src/pages/ProfileSettingsPage.jsx`, строка 191
- **Описание:** `value={editPhoneValue.length > 0 ? formatPhoneDisplay(editPhoneValue) : '+7'}` — но `formatPhoneDisplay` маскирует цифры `***`, что мешает редактированию. Input показывает маскированное значение, а onChange принимает raw digits — UX confusion.

## 3. UX проблемы

### 3.1 Критические UX проблемы

**UX-001: Нет реального подтверждения оплаты**
- После нажатия "Помочь" пользователь отправляет заявку, видит "Спасибо... формируется счёт". Но нет никакого механизма автоматической проверки оплаты. Весь процесс ручной: админ должен вручную проверить оплату в Kaspi, затем изменить статус. Это может занять часы/дни. Пользователь не получает feedback, что оплата прошла.

**UX-002: Банковская карта показана но недоступна**
- В PaymentModal радиокнопка "Банковская карта" кликабельна и можно её выбрать, но при оплате — `alert('Оплата банковской картой временно недоступна')`. Лучше визуально задизейблить.

**UX-003: Нет offline-режима**
- Приложение полностью зависит от интернета. Нет service worker, нет кэширования, нет skeleton screens (кроме спиннера). При потере сети — пустой экран.

**UX-004: Нет pull-to-refresh**
- На мобильном приложении нет возможности обновить данные жестом. Приходится переходить между экранами.

### 3.2 Средние UX проблемы

**UX-005: Выбор города через 15 секунд после загрузки — неожиданно**
- `setTimeout(15000)` для показа CitySelectionModal — пользователь может быть посередине чтения, модалка появляется неожиданно.

**UX-006: Нет индикации загрузки при отправке оплаты (подписки)**
- В модалках подписок (MainPage, FundDetailPage) кнопка "Отправить заявку" не показывает loading state, хотя идёт async запрос к Supabase.

**UX-007: Нет валидации суммы в реальном времени**
- При вводе своей суммы нет подсказки о минимуме/максимуме. Ошибка показывается только alert() при нажатии кнопки.

**UX-008: Дублирование информации между AboutFundPage и ContactsPage**
- Реквизиты фонда отображаются на обеих страницах. Путает пользователя.

**UX-009: Ссылки скачивания приложения ведут на `#`**
- Обе ссылки (Google Play, App Store) имеют `href='#'`. Бесполезные CTA.

**UX-010: Нет обратной связи при копировании ссылки**
- При Share (веб, без Share API): `alert('Ссылка скопирована в буфер обмена')` — устаревший паттерн. Лучше toast notification.

**UX-011: Нет анимации при переходах между страницами**
- SPA без page transitions. Резкие смены контента.

**UX-012: Bottom navigation не подсвечивает текущий раздел корректно**
- Город в навигации не имеет active state (всегда черный). Нет визуальной связи с текущей фильтрацией.

### 3.3 Accessibility (a11y) проблемы

**A11Y-001: Отсутствие aria-label на большинстве кнопок-иконок**
- Кнопки закрытия (X), навигации (<, >), меню — не имеют aria-label (исключение: кнопка "Поделиться" в CharityCard).

**A11Y-002: Нет skip-to-content для клавиатурной навигации**

**A11Y-003: Нет focus trap в модальных окнах**
- CharityModal, PaymentModal и другие не блокируют фокус внутри модалки.

**A11Y-004: Контраст текста**
- `text-[10px]` на категориях и суммах может быть нечитаемым для слабовидящих.
- Серый текст на белом фоне (text-secondary: #6C6C70) — граничный контраст.

**A11Y-005: Нет альтернативных текстов для декоративных изображений**
- Фоновые изображения профиля не имеют `alt=""` для screen readers.

## 4. Нагрузочное тестирование и лимиты системы

### 4.1 Архитектура и точки нагрузки

Приложение использует **Supabase Free/Pro plan** с PostgreSQL базой данных. Все данные загружаются напрямую с клиента через `@supabase/supabase-js`.

### 4.2 Supabase Free Plan лимиты

| Ресурс | Лимит Free Plan | Оценка потребления |
|--------|----------------|-------------------|
| База данных | 500 MB | Низкое (текст, JSON) |
| Storage | 1 GB | Среднее (изображения подопечных, аватары) |
| Bandwidth | 5 GB/мес | **Критично при росте** |
| API requests | 500K/мес | ~16K/день |
| Edge Functions | 500K вызовов | Низкое (только push) |
| Concurrent connections | ~200 | **Узкое место** |
| File upload size | 50 MB | Достаточно для аватаров |

### 4.3 Оценка нагрузки на один визит

Один пользователь при типичном сценарии (главная → лента → карточка → оплата → профиль) генерирует:

1. **MainPage:** 2 запроса (beneficiaries + partner_funds)
2. **HomePage:** 1 запрос (beneficiaries с фильтрами)
3. **CharityModal → PaymentModal:** 2 запроса (insert payment + upsert visitor)
4. **ProfilePage:** 4-8 запросов (visitors, payments, beneficiaries, profiles)
5. **Изображения:** ~5-15 изображений из Supabase Storage (~200KB-2MB каждое)

**Итого: ~10-15 API запросов + ~2-10 MB трафика на один визит**

### 4.4 Оценка одновременных пользователей

- **Free Plan:** ~200 concurrent connections → **~100-150 одновременных пользователей** (с учётом того что каждый держит connection pool)
- **Pro Plan ($25/мес):** ~500 concurrent → **~300-400 одновременных пользователей**
- **При 500K API requests/мес:** если средний визит = 12 запросов → **~41,000 визитов/мес** на Free Plan

### 4.5 Узкие места производительности

**PERF-001: Нет пагинации на HomePage/MainPage**
- `useMainPageData` загружает ВСЕ активные beneficiaries одним запросом. При 100+ подопечных это ~100KB JSON + все изображения.
- `useBeneficiaries` также без лимита. Только `MainPage` ограничивает `displayBeneficiaries.slice(0, 10)` на клиенте (но данные уже загружены полностью).

**PERF-002: Изображения не оптимизированы на главных страницах**
- `optimizeImage()` используется только на `FundDetailPage` и `PartnerFundsPage`. На MainPage и HomePage — оригинальные URL без трансформаций.
- Нет `srcset` / responsive images.
- Нет lazy loading на MainPage (masonry grid).

**PERF-003: ProfilePage — каскад из 5+ последовательных запросов**
- `loadProfile()` делает: visitors → visitors by phone → payments by phone → payments by visitor_ids → beneficiaries by ids → user_profiles. Все последовательно, а не параллельно. Потенциально ~500ms+ на каждом запросе.

**PERF-004: PaymentModal — аналогичный каскад при загрузке сохранённого телефона**
- `loadSavedPhone()` — 5+ последовательных запросов при каждом открытии модалки оплаты.

**PERF-005: Нет кэширования данных между навигациями**
- `dataCache.js` существует, но не используется. При каждом переходе на `/feed` данные загружаются заново.

**PERF-006: FingerprintJS загружается при каждом визите**
- `getVisitorId()` кэширует в localStorage, но `FingerprintJS.load()` всё равно вызывается при очистке кэша. Библиотека ~15KB gzip.

**PERF-007: Yandex Metrika загружается с задержкой, но блокирует main thread**
- `initYandexMetrika()` не вызывается в исходном коде App.jsx — возможно через index.html. Tag.js + WebVisor — тяжёлые скрипты.

### 4.6 Рекомендации по масштабированию

1. **Пагинация:** Добавить `.range(from, to)` в Supabase запросы с infinite scroll
2. **Image CDN:** Использовать `optimizeImage()` повсеместно, добавить `srcset`
3. **Кэширование:** Активировать `dataCache.js` для beneficiaries и funds
4. **Параллельные запросы:** В ProfilePage использовать `Promise.all()`
5. **Database index:** Убедиться что есть индексы на `category`, `city`, `is_active`, `collection_status`, `visitor_id`
6. **Upgrade:** Перейти на Supabase Pro Plan при > 100 DAU

## 5. Безопасность

### 5.1 Критические уязвимости

**SEC-001: Supabase Anon Key захардкожен в исходном коде**
- **Файл:** `/src/supabaseClient.js`, строка 4
- **Описание:** Anon key виден всем пользователям. Это нормально для Supabase (anon key предполагает RLS), но ТОЛЬКО если Row Level Security (RLS) корректно настроен на всех таблицах.
- **Риск:** Если RLS не настроен, любой может через supabase-js выполнить: `supabase.from('kaspi_payment_requests').select('*')` и получить ВСЕ платёжные данные (телефоны, суммы).
- **Необходимо проверить:** RLS policies на таблицах `kaspi_payment_requests`, `visitors`, `user_profiles`, `fund_subscriptions`, `beneficiaries`, `partner_funds`, `push_tokens`.

**SEC-002: Админ email захардкожен в клиентском коде**
- **Файл:** `/src/utils/adminAuth.js`, строка 4
- **Описание:** `const ADMIN_EMAILS = ['kleiman.alex@yandex.kz'];` — виден в bundle. Атакующий знает какие email имеют admin доступ.
- **Риск:** Целевой фишинг на admin email. Проверка должна быть на серверной стороне (Supabase RLS / Edge Function).

**SEC-003: Нет rate limiting на создание payment requests**
- **Файл:** `/src/components/PaymentModal.jsx`
- **Описание:** Защита от двойного клика через `isSubmitting`, но нет защиты от массового создания заявок (бот может создать 1000 фейковых заявок с разными телефонами).
- **Риск:** DoS на админа (завалит Kaspi запросами). Потенциально — спам номерами.

**SEC-004: Нет rate limiting на fund_subscriptions**
- Аналогично — бот может создать тысячи подписок.

**SEC-005: visitor_id в localStorage легко подделать**
- **Файл:** `/src/utils/fingerprint.js`
- **Описание:** visitor_id сначала проверяется в `localStorage`, и только при его отсутствии вычисляется через FingerprintJS. Можно подставить чужой visitor_id и видеть чужие данные в профиле.
- **Риск:** Information disclosure — телефоны и суммы пожертвований других людей.

### 5.2 Средние уязвимости

**SEC-006: Нет валидации типов файлов при загрузке аватара**
- **Файл:** `/src/pages/ProfilePage.jsx`, строка 277
- **Описание:** `accept='image/*'` только на HTML уровне. Серверная валидация зависит от Supabase Storage policies. Потенциально можно загрузить вредоносный файл.

**SEC-007: WhatsApp ссылка — потенциальный open redirect**
- **Файл:** `/src/pages/FundDetailPage.jsx`, строка 226
- **Описание:** `href={https://wa.me/${socialLinks.whatsapp.replace(/\D/g, '')}}` — если admin вставит нестандартный whatsapp, может быть XSS через `javascript:` протокол. Маловероятно при `.replace(/\D/g, '')`, но стоит проверить.

**SEC-008: Push notification data.url — open redirect**
- **Файл:** `/src/utils/pushNotifications.js`, строка 53
- **Описание:** `window.location.href = data.url;` — если push notification содержит произвольный URL, можно редиректнуть на фишинговый сайт.

**SEC-009: XSS через beneficiary description**
- React по умолчанию экранирует HTML, но `whitespace-pre-wrap` в CharityModal (строка 255) может отобразить визуально сбивающий контент (например, огромные пробелы, unicode exploits).

**SEC-010: localStorage хранит sensitive data**
- `knownPaidIds`, `adminUser`, `visitorId` — в localStorage, доступном любому JS на домене. При XSS все эти данные компрометируются.

### 5.3 Информационные

**SEC-011: Supabase URL раскрывает project ref**
- `bvxccwndrkvnwmfbfhql.supabase.co` — project reference ID. С ним можно пробовать различные endpoints.

**SEC-012: Нет Content Security Policy (CSP) headers**
- Не настроены CSP headers, что делает приложение более уязвимым к XSS.

**SEC-013: HTTP ссылки на документы**
- **Файл:** `/src/pages/DocumentsPage.jsx`, строки 14, 19, 24
- **Описание:** `http://doc.shanyrak.world/...` — HTTP, не HTTPS. Данные передаются в открытом виде. Уязвимость к MITM.
